# Технологический стек: Programming Course Platform (MVP)

Дата создания: 2026-04-16
Тип проекта: Веб-приложение (SPA + REST API)

## Выбранные технологии

### Язык программирования
Выбор: JavaScript (ES2022) для frontend, Python 3.11+ для backend
Обоснование: Текущий код проекта уже построен на React и FastAPI, поэтому этот стек сохраняет совместимость и ускоряет развитие MVP.

### Frontend
Фреймворк/Библиотека: React 18 + React Router 6
Обоснование: Подходит для SPA-навигации и role-based переходов между экранами student/teacher.

Дополнительные библиотеки:
- react 18.2.0: компонентная модель
- react-dom 18.2.0: рендеринг
- react-router-dom 6.22.3: маршрутизация

### Backend
Фреймворк: FastAPI 0.115 + Uvicorn 0.30
Обоснование: Быстрая разработка REST API, простая валидация и удобная схема для учебного проекта.

Дополнительные библиотеки:
- fastapi 0.115.0: API и валидация
- uvicorn[standard] 0.30.0: ASGI сервер

### База данных
Тип БД: SQL
Конкретная БД: SQLite 3 (файл .db)
Обоснование: По вашим обновлениям проекта целевая БД теперь SQLite. Для MVP это минимальная инфраструктура, простой запуск без отдельного сервера и достаточная функциональность для текущей нагрузки.

### Инструменты тестирования
- Unit: Pytest (backend), Vitest + React Testing Library (frontend)
- Integration: Pytest + httpx
- E2E: Playwright

### Инструменты сборки и развертывания
- Сборка frontend: Webpack 5 + Babel
- Запуск backend: Uvicorn
- Локальное развертывание: frontend dev server + backend uvicorn process

### Дополнительные инструменты
- ESLint + Prettier для frontend
- Ruff (или flake8 + black) для backend
- OpenAPI docs (автоматически через FastAPI)

## Подход к организации кода

Выбранный подход: Monorepo + feature-first + strict interfaces
Обоснование: frontend и backend работают в одном репозитории, фичи изолированы в `features/`, общий код вынесен в `packages/`, а контракты интерфейсов снижают шум и ошибки при работе с ИИ-агентом.

## Ограничения и предпочтения

- Предпочтение пользователя: использовать SQLite как БД.

## Альтернативы, которые рассматривались

### База данных
1. SQLite (выбрана)
- Плюсы: самый быстрый старт, нет отдельного DB-сервера, простой локальный запуск.
- Минусы: ограничения по конкурирующим записям и горизонтальному росту.

2. PostgreSQL
- Плюсы: выше масштабируемость и расширенные SQL-возможности.
- Минусы: лишняя инфраструктурная сложность для текущего MVP.

3. MySQL
- Плюсы: распространенность и зрелость.
- Минусы: нет явного преимущества для данного проекта относительно SQLite/PostgreSQL.

## Ключевые решения этапа

### 1. Сохранить React + FastAPI как базовый стек
- Причина: текущий код уже на этом стеке, миграция не нужна.
- Влияние: этап архитектуры будет концентрироваться на структуре модулей, а не на смене технологий.

### 2. Принять SQLite как основную БД MVP
- Причина: вы явно обновили проект под SQLite и это оптимально для локального учебного цикла.
- Влияние: архитектура должна включать слой репозиториев и схему миграции для .db файла.

### 3. Monorepo + feature-first
- Причина: снижает накладные расходы и ускоряет связку UI/API.
- Влияние: на этапе архитектуры нужно формализовать границы feature-модулей.
