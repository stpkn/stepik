## Почему этот шаг создан

Это первый шаг, потому что он покрывает приоритетную P1-функцию (Authentication) и не имеет зависимостей.
Без стабильного auth-фундамента последующие шаги по дашборду и учебным сценариям будут опираться на нестабильную основу.

## Зависимости

- **Зависит от шагов**: нет

- **Покрывает функции из концепции**:
  - **P1**: Authentication

## Ключевые решения при планировании

### РЕШЕНИЕ: Начать планирование с auth-потока, а не с UI-виджетов дашборда

- **Дата**: 2026-04-16
- **Причина**: Авторизация является обязательной точкой входа для всех защищенных экранов.
- **Альтернативы**:
  - **Альтернатива 1**: Сначала планировать Student Dashboard - не выбрана, потому что экран зависит от факта успешной авторизации.
  - **Альтернатива 2**: Сначала планировать Teacher Dashboard - не выбрана, потому что это менее базовый сценарий в MVP-потоке.
- **Влияние**: Упрощается управление зависимостями, все следующие шаги могут опираться на проверенный auth-контур.
- **Связанные решения**: `.pldf/architecture.md` (решения 2 и 3: contract-first + SQLite)

### РЕШЕНИЕ: Явно связать шаг с тремя типами артефактов (UI/API/Data)

- **Дата**: 2026-04-16
- **Причина**: Для AI-agent workflow важно исключить неоднозначность между реализацией UI и backend.
- **Альтернативы**:
  - **Альтернатива 1**: Ссылаться только на OpenAPI - не выбрана, потому что потеряется связь с UI и data моделью.
  - **Альтернатива 2**: Ссылаться только на код - не выбрана, потому что нарушается PLDF-документная трассируемость.
- **Влияние**: У каждого исполнителя есть однозначные источники истины по этому шагу.
- **Связанные решения**: `.pldf/architecture.md` (contract-first, strict interfaces)
